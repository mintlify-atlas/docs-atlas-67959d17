## Security Checklist

<Warning>
  Complete all items before deploying to production. imitsu handles sensitive secrets and requires proper security measures.
</Warning>

### Essential Security

<Steps>
  <Step title="Generate strong random secrets">
    Never use default or weak values for `IMITSU_MASTER_KEY` and `IMITSU_JWT_SECRET`.
    
    ```bash
    # Generate JWT secret
    node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
    
    # Generate master encryption key
    node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
    ```
    
    <Warning>
      If `IMITSU_MASTER_KEY` is lost, all encrypted secrets become unrecoverable. Back up this key separately from the database.
    </Warning>
  </Step>

  <Step title="Enable HTTPS">
    **Never** run imitsu over plain HTTP in production. Use one of these methods:
    
    - **nginx** with Let's Encrypt certificates
    - **Caddy** (automatic HTTPS)
    - Cloud load balancer (AWS ALB, Google Cloud Load Balancing, etc.)
    - **Cloudflare** or similar CDN with SSL
    
    See [HTTPS Setup](#https-setup) below for configuration examples.
  </Step>

  <Step title="Restrict network access">
    Limit who can reach the imitsu server:
    
    - Use firewall rules (iptables, cloud security groups)
    - Deploy in a private network/VPC
    - Use VPN or SSH tunneling for remote access
    - Whitelist trusted IP addresses
    
    **Example with ufw:**
    ```bash
    sudo ufw allow from 10.0.0.0/8 to any port 3100
    sudo ufw deny 3100
    ```
  </Step>

  <Step title="Set up regular backups">
    Back up the SQLite database regularly. See [Backup Strategy](#backup-strategy) below.
  </Step>
</Steps>

### Additional Hardening

- [ ] Run as a non-privileged user (not root)
- [ ] Use systemd security directives (`NoNewPrivileges`, `ProtectSystem`, `PrivateTmp`)
- [ ] Set file permissions: `chmod 600 .env`, `chmod 600 imitsu.db`
- [ ] Enable audit logging and monitor for suspicious activity
- [ ] Implement rate limiting at the reverse proxy level (nginx `limit_req`)
- [ ] Keep Node.js and dependencies up to date
- [ ] Use `NODE_ENV=production` to disable debug features
- [ ] Disable unnecessary services on the host
- [ ] Configure log rotation to prevent disk space issues

## HTTPS Setup

### Option 1: nginx with Let's Encrypt

<Steps>
  <Step title="Install nginx and certbot">
    ```bash
    sudo apt update
    sudo apt install nginx certbot python3-certbot-nginx
    ```
  </Step>

  <Step title="Configure nginx">
    Create `/etc/nginx/sites-available/imitsu`:
    
    ```nginx
    server {
        listen 80;
        server_name secrets.company.com;
        
        # Let's Encrypt challenge
        location /.well-known/acme-challenge/ {
            root /var/www/html;
        }
        
        # Redirect to HTTPS
        location / {
            return 301 https://$server_name$request_uri;
        }
    }
    
    server {
        listen 443 ssl http2;
        server_name secrets.company.com;
        
        ssl_certificate /etc/letsencrypt/live/secrets.company.com/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/secrets.company.com/privkey.pem;
        
        # Mozilla SSL Configuration Generator (Intermediate)
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';
        ssl_prefer_server_ciphers off;
        
        # HSTS
        add_header Strict-Transport-Security "max-age=63072000" always;
        
        # Rate limiting
        limit_req_zone $binary_remote_addr zone=imitsu:10m rate=10r/s;
        limit_req zone=imitsu burst=20 nodelay;
        
        location / {
            proxy_pass http://localhost:3100;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            
            # Timeouts
            proxy_connect_timeout 60s;
            proxy_send_timeout 60s;
            proxy_read_timeout 60s;
        }
    }
    ```
    
    Enable the site:
    ```bash
    sudo ln -s /etc/nginx/sites-available/imitsu /etc/nginx/sites-enabled/
    sudo nginx -t
    sudo systemctl reload nginx
    ```
  </Step>

  <Step title="Obtain SSL certificate">
    ```bash
    sudo certbot --nginx -d secrets.company.com
    ```
    
    Certbot automatically configures nginx and sets up auto-renewal.
  </Step>

  <Step title="Test auto-renewal">
    ```bash
    sudo certbot renew --dry-run
    ```
  </Step>
</Steps>

### Option 2: Caddy (Automatic HTTPS)

<Steps>
  <Step title="Install Caddy">
    ```bash
    sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https
    curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
    curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
    sudo apt update
    sudo apt install caddy
    ```
  </Step>

  <Step title="Configure Caddyfile">
    Edit `/etc/caddy/Caddyfile`:
    
    ```caddy
    secrets.company.com {
        reverse_proxy localhost:3100
        
        # Rate limiting
        rate_limit {
            zone dynamic {
                key {remote_host}
                events 100
                window 1m
            }
        }
    }
    ```
  </Step>

  <Step title="Start Caddy">
    ```bash
    sudo systemctl enable caddy
    sudo systemctl start caddy
    ```
    
    Caddy automatically obtains and renews Let's Encrypt certificates.
  </Step>
</Steps>

### Option 3: Fly.io (Managed HTTPS)

imitsu includes a `fly.toml` configuration for Fly.io deployment:

```toml
app = 'imitsu'
primary_region = 'lhr'

[build]

[env]
  IMITSU_DB_PATH = '/data/imitsu.db'
  IMITSU_PORT = '3100'
  NODE_ENV = 'production'

[[mounts]]
  source = 'imitsu_data'
  destination = '/data'

[http_service]
  internal_port = 3100
  force_https = true
  auto_stop_machines = 'suspend'
  auto_start_machines = true
  min_machines_running = 1

[[vm]]
  size = 'shared-cpu-1x'
  memory = '512mb'
```

<Steps>
  <Step title="Install flyctl">
    ```bash
    curl -L https://fly.io/install.sh | sh
    ```
  </Step>

  <Step title="Set secrets">
    ```bash
    fly secrets set IMITSU_JWT_SECRET="your-jwt-secret"
    fly secrets set IMITSU_MASTER_KEY="your-master-key"
    ```
  </Step>

  <Step title="Deploy">
    ```bash
    fly deploy
    ```
    
    Fly.io automatically provides HTTPS and a `.fly.dev` domain.
  </Step>
</Steps>

## Backup Strategy

### Why Backups Are Critical

- The SQLite database (`imitsu.db`) contains all encrypted secrets, users, teams, and audit logs
- If `IMITSU_MASTER_KEY` is lost, encrypted secrets cannot be recovered
- Database corruption or accidental deletion can cause data loss

### Backup Frequency

- **Minimum:** Daily backups
- **Recommended:** Hourly backups with retention policy
- **Critical systems:** Real-time replication + hourly snapshots

### Backup Methods

#### Manual Backup

```bash
# Stop the server (recommended)
sudo systemctl stop imitsu

# Copy the database
cp /opt/imitsu/data/imitsu.db /backup/imitsu-$(date +%Y%m%d-%H%M%S).db

# Restart the server
sudo systemctl start imitsu
```

#### Online Backup (No Downtime)

Using SQLite's online backup API:

```bash
sqlite3 /opt/imitsu/data/imitsu.db ".backup '/backup/imitsu-$(date +%Y%m%d-%H%M%S).db'"
```

#### Automated Backup Script

Create `/usr/local/bin/backup-imitsu.sh`:

```bash
#!/bin/bash
set -euo pipefail

BACKUP_DIR="/backup/imitsu"
DB_PATH="/opt/imitsu/data/imitsu.db"
RETENTION_DAYS=30

mkdir -p "$BACKUP_DIR"

# Create backup
BACKUP_FILE="$BACKUP_DIR/imitsu-$(date +%Y%m%d-%H%M%S).db"
sqlite3 "$DB_PATH" ".backup '$BACKUP_FILE'"

# Compress
gzip "$BACKUP_FILE"

# Delete old backups
find "$BACKUP_DIR" -name "imitsu-*.db.gz" -mtime +$RETENTION_DAYS -delete

echo "Backup completed: $BACKUP_FILE.gz"
```

Make it executable and add to cron:

```bash
sudo chmod +x /usr/local/bin/backup-imitsu.sh
sudo crontab -e
```

Add hourly backups:

```cron
0 * * * * /usr/local/bin/backup-imitsu.sh >> /var/log/imitsu-backup.log 2>&1
```

#### Docker Backup

```bash
# Named volume
docker run --rm \
  -v imitsu-data:/data \
  -v $(pwd)/backup:/backup \
  alpine sh -c "cp /data/imitsu.db /backup/imitsu-$(date +%Y%m%d-%H%M%S).db"

# Bind mount
cp /opt/imitsu/data/imitsu.db /backup/imitsu-$(date +%Y%m%d-%H%M%S).db
```

### Backup Verification

Regularly test your backups:

```bash
# Check database integrity
sqlite3 backup-20260306.db "PRAGMA integrity_check;"

# List tables
sqlite3 backup-20260306.db ".tables"

# Count records
sqlite3 backup-20260306.db "SELECT COUNT(*) FROM secrets;"
```

### Off-site Backups

Store backups in multiple locations:

- **Amazon S3:** Use `aws s3 cp` or `rclone`
- **Backblaze B2:** Use `rclone`
- **rsync to remote server:** `rsync -avz /backup user@remote:/backup/imitsu`

**Example with rclone to S3:**

```bash
rclone copy /backup/imitsu s3:my-bucket/imitsu-backups --include "*.db.gz"
```

### Restore Procedure

<Steps>
  <Step title="Stop the server">
    ```bash
    sudo systemctl stop imitsu
    ```
  </Step>

  <Step title="Restore the database">
    ```bash
    # Decompress if needed
    gunzip backup-20260306.db.gz
    
    # Restore
    cp backup-20260306.db /opt/imitsu/data/imitsu.db
    sudo chown imitsu:imitsu /opt/imitsu/data/imitsu.db
    ```
  </Step>

  <Step title="Verify integrity">
    ```bash
    sqlite3 /opt/imitsu/data/imitsu.db "PRAGMA integrity_check;"
    ```
  </Step>

  <Step title="Start the server">
    ```bash
    sudo systemctl start imitsu
    sudo systemctl status imitsu
    ```
  </Step>
</Steps>

## Monitoring

### Health Checks

The server runs on port 3100. Check if it's responding:

```bash
curl -f http://localhost:3100 || echo "Server down"
```

### Systemd Status

```bash
sudo systemctl status imitsu
sudo journalctl -u imitsu -f
```

### Docker Health Check

Add to `docker-compose.yml`:

```yaml
healthcheck:
  test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:3100"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 40s
```

### Monitoring Tools

- **Uptime Kuma:** Self-hosted monitoring
- **Prometheus + Grafana:** Metrics and dashboards
- **Cloud monitoring:** AWS CloudWatch, Google Cloud Monitoring, Datadog

### Audit Log Monitoring

Regularly review the audit log:

```bash
imitsu audit -l 100
```

Alert on suspicious activity:
- Multiple failed login attempts
- Unexpected secret deletions
- Access from unknown IPs

## Performance Tuning

### SQLite Optimization

The database is automatically configured with optimal settings, but you can manually optimize:

```bash
sqlite3 /opt/imitsu/data/imitsu.db "VACUUM;"
sqlite3 /opt/imitsu/data/imitsu.db "ANALYZE;"
```

### Rate Limiting

imitsu has built-in rate limiting (100 requests/minute per IP). Add additional rate limiting at the reverse proxy level for defense in depth.

### Resource Limits

**systemd:**

```ini
[Service]
MemoryLimit=512M
CPUQuota=50%
```

**Docker:**

```yaml
services:
  imitsu:
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
```

## Troubleshooting

### Check logs

```bash
sudo journalctl -u imitsu -n 100 --no-pager
```

### Database locked errors

- Ensure only one instance is accessing the database
- Check for long-running transactions
- Increase SQLite timeout (requires code change)

### Permission denied

```bash
sudo chown -R imitsu:imitsu /opt/imitsu/data
sudo chmod 600 /opt/imitsu/data/imitsu.db
```

### High memory usage

- Check for large audit logs: `sqlite3 imitsu.db "SELECT COUNT(*) FROM audit_log;"`
- Consider archiving old audit entries
